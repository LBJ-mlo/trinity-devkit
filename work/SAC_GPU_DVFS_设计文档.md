## 1. 问题定义与目标

### 1.1 场景描述

大模型推理（LLM Serving）在 GPU 上运行时，NVIDIA 默认的 DVFS（Dynamic Voltage and Frequency Scaling）策略以"性能最大化"为目标，尽可能维持高频率。然而，LLM 推理并不总是需要最高频率——在很多情况下，降低频率对吞吐（TPS）影响极小，却能大幅降低功耗。

### 1.2 优化目标

> **在有约束的条件下最小化 GPU 能耗**
>
> - **优化目标**：最小化 GPU 能耗（Joules）
> - **硬约束**：TPS ≥ SLO_target（如 50 tokens/s）
> - **软约束**：TTFT ≤ 500ms、避免频率震荡、避免温度过高

### 1.3 为什么这个问题适合 RL

| 问题特征 | 为什么传统方法不够 | 为什么 RL 合适 |
|---------|-------------------|---------------|
| 负载多变（QPS波动、prompt长短不一） | PID 只能对当前误差做反应，无法预测 | RL 可以学习"状态→最优频率"的映射 |
| Prefill/Decode 两阶段的物理特性完全不同 | 手工规则难以覆盖所有组合 | RL 可以从状态中自动识别阶段特征 |
| "频率-功耗-TPS"是非线性的 U 型关系 | 需要显式建模，且模型随负载变化 | RL 通过奖励信号隐式学习 |
| 多目标权衡（节能 vs TPS vs 温度 vs 稳定性） | 需要手工加权 | SAC 的奖励函数自然融合多目标 |
| 缺乏精确的物理模型 | 传统控制（如MPC）需要系统模型 | SAC 是 model-free，不需要先验模型 |

### 1.4 关键指标

| 指标 | 定义 | 目标 |
|------|------|------|
| **平均功耗** | GPU 平均功率（W） | 比默认 DVFS 降低 ≥20% |
| **总能耗** | 功耗 × 时间积分（J） | 比默认 DVFS 降低 ≥20% |
| **TPS SLO 达标率** | `TPS ≥ SLO_target` 的时间占比 | ≥99% |
| **频率震荡率** | 频率切换次数/分钟 | <20 次/分钟（避免来回抖动） |
| **SLO 违例恢复时间** | TPS 从跌破 SLO 到恢复的时间 | <500ms |

## 2. 系统原理与核心机制

### 2.1 LLM 推理的两阶段物理特性

LLM 推理包含两个物理特性截然不同的阶段，这是整个系统设计最底层的依据。

```
Prefill 阶段（计算密集型）:
  ┌────────────────────────────────┐
  │ 并行处理所有 prompt tokens     │
  │ 矩阵-矩阵乘法（GEMM）          │
  │ 算术强度: 55-100 FLOP/Byte     │
  │ GPU 利用率: ~90%               │
  │ 功耗: 高（接近 TDP）           │
  │ 频率敏感: 高 ← 降频直接增加 TTFT │
  │ 每请求能量占比: 小（执行时间短） │
  └────────────────────────────────┘

Decode 阶段（内存密集型）:
  ┌────────────────────────────────┐
  │ 逐 token 生成                  │
  │ 矩阵-向量乘法（GEMV）          │
  │ 算术强度: 1-10 FLOP/Byte       │
  │ GPU 利用率: 15-35%             │
  │ 功耗: 低（仅 20-40% TDP）      │
  │ 频率敏感: 极低 ← 降频几乎不影响 TPS │
  │ 每请求能量占比: 大（执行时间长） │
  └────────────────────────────────┘
```

**关键推论（直接影响设计决策）**：

1. **Decode 是节能的主战场**（能量占比大），而且**对频率不敏感**——这意味着可以大幅降频而 TPS 几乎不降
2. **Prefill 需要精细控制**（频率显著影响 TTFT），但能量贡献相对小
3. 频率-功耗-性能曲线呈 **U 型**：频率太低→执行时间拉长→总能量不降反升。存在一个"最优频率点"

> **数据支撑**（来自 GreenLLM DAC'26、UIUC H100 实测）：
> - A100 上 Qwen-14B/30B：通过分阶段 DVFS 总能耗降 34%，SLO 违例增加 <3.5%
> - H100 上 Llama-2 70B：频率从 2GHz→1.6GHz，TPS 几乎不变，能耗降 ~20%
> - Decode 频率降至 40% max 时 TPS 仅降 ~5-10%，但功耗降 ~60%+

### 2.2 为什么需要 RL 而不是 PID

PID 控制器的本质是一维误差反馈：

```text
PID:  error = TPS_target - TPS_observed
      freq = Kp×error + Ki×∫error + Kd×d(error)/dt
```

PID 的局限：
- **只能对一个变量做反应**（TPS误差），无法利用 GPU 利用率、队列深度、prompt长度分布等丰富的状态信息
- **是被动的**——只有在 TPS 已经变化了才能响应，无法预测负载即将变化
- **参数是固定的**——当流量 pattern 改变（如从短 prompt 为主变为长 prompt 为主），需要重新调 Kp/Ki/Kd

SAC 的优势：
- **多维度状态感知**：同时观察 GPU 利用率、内存带宽利用率、功耗、温度、队列深度、TPS 趋势——自动发现"现在该提频"的多维信号组合
- **可学习预测性决策**：通过奖励信号学到"队列深度上升 + GPU利用率攀升 → 即将需要更高频率"的预测性策略
- **自动适应**：在线持续训练可以适应流量 pattern 变化
- **自然处理非线性**：U 型曲线的发现、多目标权衡，不需要显式建模

### 2.3 系统的控制哲学：双层结构

```text
┌──────────────────────────────────────────────┐
│  SAC: 智能层（2s 周期）                       │
│  负责: 在正常情况下找到最优频率                │
│  特点: 可学习、可犯错、可优化                  │
│  输入: 15维状态 → 输出: 目标频率               │
└──────────────────┬───────────────────────────┘
                   │ target_freq
                   ▼
┌──────────────────────────────────────────────┐
│  SLA Guardian: 安全层（100ms 周期）            │
│  负责: TPS 跌破安全线时强制提频，覆盖 SAC      │
│  特点: 纯规则、不可学习、不可绕过              │
│  逻辑: IF TPS < 85%×SLO → freq = f_max       │
└──────────────────┬───────────────────────────┘
                   │ final_freq
                   ▼
┌──────────────────────────────────────────────┐
│  GPU 硬件                                     │
│  实际执行频率，同时自带 thermal throttling     │
└──────────────────────────────────────────────┘
```

**设计依据**：SAC 需要探索才能学到好策略，但探索意味着可能犯错。Guardian 让 SAC 在安全围栏内自由探索——SAC 犯了错，惩罚会通过奖励函数传导给学习过程，但实际频率不会被拉到危险水平。

这类似于工业控制中的"内环-外环"架构：内环（Guardian）保证安全，外环（SAC）追求优化。

## 3. Why SAC：算法选型论证

### 3.1 强化学习算法对比

| 算法 | 类型 | 动作空间 | 样本效率 | 探索机制 | 对本问题的适配 |
|------|------|---------|---------|---------|--------------|
| **DQN** | Off-policy | 离散 | 高 | ε-greedy | ❌ GPU频率是连续域，离散化会丢失精度 |
| **DDPG** | Off-policy | 连续 | 高 | 确定性+噪声 | ⚠️ 确定性策略容易收敛到局部最优 |
| **TD3** | Off-policy | 连续 | 高 | 确定性+噪声 | ⚠️ 改善了DDPG但仍缺乏结构化探索 |
| **PPO** | On-policy | 连续 | 低 | 随机策略 | ❌ 每个频率调整只产生一个样本，on-policy太浪费 |
| **SAC** | **Off-policy** | **连续** | **高** | **最大熵** | ✅ 三项特性全部命中 |

### 3.2 SAC 的三个核心机制

**机制1：最大熵策略**

```text
标准RL目标:  max Σ E[r(s,a)]
SAC目标:    max Σ E[r(s,a) + α·H(π(·|s))]

H(π(·|s)) = 策略的熵，即"策略有多随机"

α (温度参数) 控制了探索的程度:
  - 训练初期: α大 → 鼓励探索，广泛尝试各种频率
  - 训练后期: α减小 → 策略收敛，动作变得确定
```

**对本场景的价值**：频率的选择空间很大（500~1980MHz），且最优频率在不同负载下不同。最大熵探索确保 SAC 不会过早收敛到某个看起来不错但并非最优的频率。

**机制2：Clipped Double-Q（双Q网络）**

```text
SAC 维护两个 Critic 网络 Q1, Q2
目标 Q 值 = min(Q1(s,a), Q2(s,a))

目的: 抑制 Q 值的过估计（overestimation bias）
      两个网络面对同样状态-动作时取较小者
```

**对本场景的价值**：频率调整的奖励信号有噪声（TPS 测量有抖动），双Q 能防止 SAC 被偶然的高奖励误导，误以为某个频率特别好。

**机制3：自动温度调节（Auto α）**

```text
α 的更新目标:
  min E[-α·(log π(a|s) + H_target)]

H_target = -dim(action) = -1 (单维动作)

意义: α 自动调整，不需要手动调这个超参数
      当策略不确定性大时自动增加探索
      当策略开始收敛时自动减少探索
```

**对本场景的价值**：不需要猜测"探索到什么程度合适"。在不同负载 pattern 下，SAC 自动判断还需要多少探索。

### 3.3 Off-policy 对本场景的特殊意义

```text
On-policy (PPO):
  收集数据 → 用这批数据更新一次 → 丢弃数据 → 重新收集
  GPU调频: 每2s才一个样本, 一天才43200个

Off-policy (SAC):
  所有历史样本存入 Replay Buffer → 每次更新随机采样
  几天前的"高负载下提频"经验今天仍然有用
  支持"白天在线运行, 晚上离线大规模训练"
```

**结论**：SAC 是唯一同时满足"连续动作 + Off-policy高样本效率 + 结构化探索"的成熟算法。选择 SAC 不是偏好，是约束下的最优解。

## 4. MDP设计

MDP 设计决定了 SAC 能学到什么、学得多快、学到的东西对不对。这是整个系统最核心的设计环节。

### 4.1 状态空间（15维）

状态空间的设计原则：**包含所有影响"最优频率"的因素，排除无关或冗余信息**。

> **v1.1 更新说明**：由 14 维扩展到 15 维，新增 `s₀`（当前 batch 的平均输出 token 需求），用于让 SAC 感知 Decode 阶段的预期持续时间——这决定了节能策略的"时间窗口"有多大。

```text
状态向量 State ∈ ℝ¹⁵:

┌──────────────────────────────────────────────────────────┐
│ 组1: 推理负载特征（5维）—— 来自 vLLM metrics / 调度器      │
├──────────────────────────────────────────────────────────┤
│ s₀: 当前 batch 中平均 prompt token 数（归一化到 [0,1]）    │
│     → 区分 Prefill/Decode："当前是否在处理大段输入"        │
│     → 来源: vLLM /metrics 端点 的 prompt_tokens 统计       │
│                                                          │
│ s₁: 当前 batch 平均期望输出 token 数（归一化到 [0,1]）    │
│     → ★ 关键新增特征: "请求还会在 Decode 待多久？"        │
│     → 输出长 = Decode 时间长 = 节能窗口大                │
│     → 来源: 每个请求的 max_tokens 参数求均值              │
│     → 为什么重要:                                        │
│       - 输出 50 tokens: 请求 3 秒结束, SAC 调一次频       │
│         来不及精细优化就没了                              │
│       - 输出 1500 tokens: 请求 90 秒不结束,               │
│         SAC 可以大幅降频并维持很久                        │
│       - 输出 5000 tokens+: 请求 5 分钟,                   │
│         最优策略可能是"降到极低频率"                      │
│     → 归一化: output_avg / max_output_capacity            │
│                                                          │
│ s₂: batch_size（当前正在处理的请求数，归一化）             │
│     → batch 越大，GPU 并行度越高，频率利用率越高          │
│     → 来源: vLLM num_requests_running                     │
│                                                          │
│ s₃: queue_depth（等待队列中的请求数，归一化）             │
│     → 队列深=未来负载高，需要提前提频                    │
│     → 来源: vLLM num_requests_waiting                     │
│                                                          │
│ s₄: prompt 长度变异系数（CV = std/mean）                  │
│     → 负载多样性高时，策略应该更保守                     │
│     → 来源: 从 s₀ 的滑动窗口计算                          │
├──────────────────────────────────────────────────────────┤
│ 组2: GPU 硬件状态（5维）—— 来自 NVML                      │
├──────────────────────────────────────────────────────────┤
│ s₅: GPU SM 利用率（[0,1]）                                │
│     → 最直接的"忙不忙"指标                               │
│     → 来源: nvmlDeviceGetUtilizationRates().gpu           │
│                                                          │
│ s₆: 显存带宽利用率（[0,1]）                               │
│     → 与 s₄ 的比值区分计算密集 vs 内存密集               │
│     → gpu_util↑ + mem_util↓ = 计算密集 (Prefill)          │
│     → gpu_util↓ + mem_util↑ = 内存密集 (Decode)           │
│     → 来源: nvmlDeviceGetUtilizationRates().memory        │
│                                                          │
│ s₇: 当前功耗（归一化，以 TDP 为基准）                    │
│     → 功耗=频率×电压²+静态功耗                           │
│     → 来源: nvmlDeviceGetPowerUsage() / 1000 (W)          │
│                                                          │
│ s₈: GPU 温度（归一化到 [0,1]）                           │
│     → 高温限制降频空间                                   │
│     → 来源: nvmlDeviceGetTemperature(..., GPU)            │
│                                                          │
│ s₉: 当前 SM 频率（归一化到 [0,1]）                       │
│     → 上一动作的结果，提供时序连续性                     │
│     → 来源: nvmlDeviceGetClockInfo(..., SM)               │
├──────────────────────────────────────────────────────────┤
│ 组3: SLO 满足情况（4维）—— 计算得出                      │
├──────────────────────────────────────────────────────────┤
│ s₁₀: 当前 TPS / TPS_SLO（达标率）                          │
│     → >1.0 = 超额完成, <1.0 = 违例                       │
│     → 来源: 从最近 500ms 的 token 生成时间戳计算          │
│                                                          │
│ s₁₁: TPS 短欠比 max(0, 1 - TPS/SLO)                      │
│     → 只关注"不足"，忽略"超额"                           │
│     → 让 SAC 明确感知违例的严重程度                       │
│                                                          │
│ s₁₂: TPS 趋势（最近 10 个观测点的线性回归斜率）           │
│     → >0 = TPS 在上升, <0 = TPS 在下降                   │
│     → 下降趋势出现时就应该警惕，不用等到跌破 SLO         │
│     → 来源: 从 TPS 历史滑动窗口计算                       │
│                                                          │
│ s₁₃: 频率切换是否已完成（0=未完成, 1=已完成）            │
│     → DVFS 切换有 ~100ms 延迟                            │
│     → 如果上次设置还没生效，SAC不应再次调整               │
│     → 来源: 记录上次 nvmlDeviceSetApplicationsClocks 的时间│
├──────────────────────────────────────────────────────────┤
│ 组4: 历史记忆（1维）                                     │
├──────────────────────────────────────────────────────────┤
│ s₁₄: 最近是否被 SLA Guardian 干预过（0/1）               │
│     → "上次因为太激进被 Guardian 纠正了"                 │
│     → 让 SAC 记住教训，在类似状态下收敛到安全区域         │
│     → 来源: SLA Guardian 设置标志位                       │
└──────────────────────────────────────────────────────────┘
```

**维度选择依据**：

- **15 维对 SAC 来说是轻量的**（标准 SAC 可以处理几十到上百维的状态）
- **每个维度都有明确的物理含义**，不是黑盒特征
- **不需要手动设计如何组合这些特征**——SAC 的神经网络会自己学到非线性组合
- **信息不冗余**：利用率、TPS、功耗测量的是不同层面的东西

**⚠️ Continuous batching 下的状态采集注意事项**：

vLLM 使用 continuous batching——batch 组成每几十毫秒就可能变一次（新请求加入、完成请求退出）。但是 SAC 以 2s 固定间隔采样。

如果 s₀（prompt_length_avg）和 s₁（output_length_avg）用的是采集时刻的瞬时值，它们可能不代表过去 2s 内的"平均水平"。

**解决方案**：所有 batch 级特征（s₀, s₁, s₂, s₃）都使用**采样周期内的滑动平均**而非瞬时 snapshot：

```text
s₀ = avg(prompt_lengths of all completions in last 2s)
      / max_prompt_len
s₁ = avg(max_tokens of requests in last 2s)
      / max_output_capacity
s₂ = avg(batch_size of last 10 NVML samples)
      / max_batch_capacity
s₃ = avg(queue_depth of last 10 NVML samples)
      / max_queue_capacity
```

实现方式：在 2s 周期内每 200ms 取一次快照记录到环形缓冲区，step 触发时取平均值。

### 4.2 动作空间（1维连续）

```text
动作空间 A ∈ ℝ¹:

a ∈ [-1, 1]

映射到实际频率:
  target_freq = f_min + (a + 1) / 2 × (f_max - f_min)
  
  例: f_min=500MHz, f_max=1980MHz, a=0.0
      → target_freq = 500 + 0.5 × 1480 = 1240 MHz

量化: target_freq → round(target_freq / 15) × 15 MHz
       (匹配 GPU Boost 的 15MHz 步进粒度)
```

**为什么是连续而非离散**：
- GPU 频率本质是连续物理量（虽然硬件上量化到 15MHz 步进）
- 离散化（如分成 10 个 P-State）会丢失精度——最优频率可能在两个离散点之间
- SAC 的 tanh 策略天然输出 [-1,1]，直接映射到频率范围，无需额外处理

**为什么是单维而非多维**：
- 只控制 SM 频率——显存频率通常保持固定（decode 是内存密集的，降显存频率会直接伤害 TPS）
- 电压由 GPU 硬件根据频率自动调节，不可编程

**GPU 类型与 NVML 接口可用性**：

不同 GPU 支持的频率控制接口不同，实施前必须确认目标硬件：

```text
数据中心 GPU (A100, H100, H200):
  ✅ nvmlDeviceSetApplicationsClocks(handle, mem_clock, sm_clock)
  → 设置应用时钟上限，GPU 实测不会超过该时钟
  → 可用 nvidia-smi -ac 验证
  → 恢复默认: nvmlDeviceResetApplicationsClocks(handle)
  → 注意: SM 时钟不能超过当前设置的显存时钟对应的最大 SM 时钟

Jetson 嵌入式系列 (Orin, Xavier):
  ✅ nvmlDeviceSetGpuLockedClocks(handle, min, max)
  → 锁定 SM 频率范围, 不设置 app clocks
  → 可用 /sys/devices/gpu.0/devfreq/... 验证

GeForce/RTX 系列:
  ❌ nvmlDeviceSetApplicationsClocks 通常返回 NVMLError_NOT_SUPPORTED
  ✅ 备用方案 1: nvidia-smi --lock-gpu-clocks=min,max
     (Linux 上需要 persistence mode 开启)
  ✅ 备用方案 2: nvidia-smi -lgc min,max
  → 如果两个方案都不可用（如锁定驱动或 Windows），
    SAC 只能做"建议"而无法实际控制频率
  → 解决方案: 在 Docker 容器中运行并启用
    NVIDIA_REQUIRE_* 环境变量

通用降级方案（所有 GPU）:
  nvidia-smi -pl <power_limit>
  → 设置功率上限 → GPU 驱动自动降频匹配该功率
  → 虽然不是直接频率控制, 但同样能实现"降频节能"的效果
  → 代价: 控制粒度更粗, 反应更慢
```

**实施前必须做的验证**：
```text
1. 在目标 GPU 上运行: nvidia-smi -h | grep "lock-gpu\|applications"
2. 验证: python -c "import pynvml; pynvml.nvmlInit(); h=pynvml.nvmlDeviceGetHandleByIndex(0); print(pynvml.nvmlDeviceSetApplicationsClocks(h, 5000, 1000))"
3. 如果都不可用, 备选方案: 用 nvidia-smi -pl 限制功率上限
```

### 4.3 奖励函数

奖励函数决定 SAC 学什么。这是最需要仔细设计的部分——**一个设计不好的奖励函数比一个不好的网络结构更致命**。

```text
奖励函数设计原则:
  1. SLO 惩罚 >> 节能奖励（量级差 ≥10倍）
  2. 二次惩罚（小违例被容忍，大违例不可接受）
  3. TPS "过高"时也有轻微惩罚（不需要跑满频率）
  4. 频率切换有代价（防止震荡）
  5. 奖励值域控制在 [-100, 10] 范围内

奖励计算逻辑:
┌─────────────────────────────────────────────┐
│                                             │
│  IF current_tps < 0.7 × tps_slo:            │
│    → reward = -100.0                        │
│    （严重违例: 直接给最大惩罚）              │
│                                             │
│  ELIF current_tps < tps_slo:                │
│    → violation = (tps_slo - tps) / tps_slo  │
│    → reward = -10.0 × violation²            │
│    （轻度违例: 二次惩罚，越严重惩罚越重）    │
│                                             │
│  ELSE:  (TPS ≥ SLO，满足约束)              │
│    → 节能奖励:                              │
│      energy_saving = (avg_power - power)/    │
│                       avg_power              │
│      reward += 10.0 × energy_saving          │
│      avg_power = 最近 2s (整 step 周期) 的    │
│      运行平均功耗, 非瞬时 snapshot           │
│      → 避免单次采样噪声误导 SAC 决策         │
│                                             │
│    → TPS 过度满足惩罚:                       │
│      margin = (tps - tps_slo) / tps_slo     │
│      reward += -0.2 × margin                │
│      （TPS 超标说明频率还能降，催促 SAC 试探） │
│                                             │
│    → 频率切换代价:                           │
│      freq_delta = abs(new_freq - old_freq)/  │
│                    max_freq                  │
│      reward += -0.5 × freq_delta            │
│      （抑制不必要的频率调整）                │
│                                             │
└─────────────────────────────────────────────┘
```

**为什么惩罚用二次函数（violation²）**：

```text
线性惩罚: reward = -10 × violation
  违例 5%:  -0.5  ─┐
  违例 50%: -5.0   ├─ 比例关系，10倍违例=10倍惩罚
                    │   SAC 可能觉得"20%违例没关系"
二次惩罚: reward = -10 × violation²
  违例 5%:  -0.025 ─┐
  违例 50%: -2.5    ├─ 违例翻倍，惩罚翻4倍
                    │   SAC 学到"小违例可容忍，大违例必须避免"
```

**为什么有"频率切换代价"**：
- 频率切换本身有开销（~100ms 的切换延迟期间 GPU 可能在过渡状态）
- 来回震荡（如 500→1980→500→1980...）不仅无益，还浪费能量
- 这个惩罚项让 SAC 学会"做了决定就保持一阵，不要频繁抖动"

### 4.4 Episode 终止条件（done 信号）

标准 SAC 的 MDP 需要有 `done` 信号——表示一个 episode 结束。但对于持续的 GPU 调频任务，**没有自然的 episode 边界**：没有"关卡通关"、"任务完成"的概念，系统是持续运行的。

**当前方案**：

```text
done = False 始终成立（无限 horizon）

SAC 使用 γ < 1 处理无限 horizon:
  - 累积回报 = Σ(γ^t × r_t) 在 t → ∞ 时收敛（因为 γ < 1）
  - 不需要 done 信号也能良好地收敛
  - 在持续运行场景下这是标准做法

唯一例外:
  - 如果负载注入脚本的某一阶段结束、准备切换到下一阶段时，
    可以用 done=True + 重置 replay buffer —— 但这通常没必要
  - 只在以下情况考虑: 负载特征在阶段间剧烈变化，旧策略完全
    不适用于新场景时（我们用了缓过渡，所以不需要）
```

**为什么不需要人为构造 episode 边界**：
```text
有人可能想"把每5分钟当作一个episode, 然后在结束时给一个
全局效率评估作为terminal reward"。这在训练时会分化:
  - γ 用在 5 分钟 × 30 steps/min = 150 步后, γ^150 = 0.0005
  - terminal reward 在 150 步后几乎不被考虑
  - 尝试构造 episode 反而引入了一个不需要的超参数

更简洁的做法: 不用 done, 让 SAC 在无限 horizon 下训练。
```

### 4.5 折扣因子 γ

```text
γ = 0.95

为什么不是 0.99:
  - 0.99 适用于"动作影响数百步后"的场景（如游戏）
  - GPU 调频一步的影响主要在未来 5-10 步（10-20秒内）
  - 更高的 γ 会让 SAC 过分关注远期影响，而当前场景的远期不确定性很高
```

## 5. 控制回路设计

### 5.1 Step 间隔：2秒

```text
Step 间隔 = 2s

选择依据:
┌────────────────────────────────────────────────────┐
│ 1. 频率切换生效需要 ~100-150ms                     │
│ 2. 新频率需要 ~5个token 才能稳定影响TPS            │
│    对于 30B+ 模型，单 token 约 60-120ms             │
│    所以 TPS 稳定在新频率需要 5×80ms = 400ms         │
│ 3. 稳定后还需要 ~3-5个token 来准确测量TPS          │
│    所以从设频率到能准确评估效果 = 400+320 = ~720ms  │
│ 4. ×2的安全余量 = 1.5s                             │
│ 5. 取整到 2s                                       │
│                                                    │
│ 适用模型: 13B-70B+                                 │
│ 小模型(7B): 可缩短到 500ms-1s                       │
└────────────────────────────────────────────────────┘
```

### 5.2 控制循环：固定定时器，非事件驱动

```text
为什么用固定间隔而不是事件驱动:

❌ 请求完成事件驱动:
  - "请求已经处理完了"才调频，对已完成的请求无意义
  - 请求完成的时间点在 Continuous Batching 下随机分布
  - 无法保证稳定的采样节奏

❌ Token 生成事件驱动:
  - Token 生成太频繁（几十ms一个），控制滞后
  - 单个 token 的 TPS 波动太大，噪声淹没信号

✅ 固定 2s 定时器:
  - 稳定节拍，每次观测到的是"过去2s的状态+性能"
  - 与请求边界无关，始终在调整当前和未来的行为
  - 每一轮 SAC 决策影响的是"接下来2s内被处理的token和请求"
```

### 5.3 控制回路架构

```text
Main Thread:
  while True:
    sleep(2s)
    state = collect_state()        # 采集15维状态
    action = sac_actor(state)      # SAC推理 (<1ms)
    target_freq = map(action)      # [-1,1] → [f_min, f_max]
    
    # 写入 transition 到 replay buffer (异步训练用)
    buffer.push(prev_state, prev_action, 
                calc_reward(), state, done=False)
    
    # 实际调频
    safe_freq = sla_guardian.check(target_freq)
    set_gpu_freq(safe_freq)
    
    prev_state = state
    prev_action = action

  注意: 第一次循环时 prev_state=None,
  跳过 buffer.push (用 if prev_state is not None 保护)
  这仅跳过第一个 iteration, 不影响后续

SLA Guardian Thread (独立, 100ms):
  while True:
    sleep(0.1s)
    if current_tps < 0.85 * tps_slo:
      set_gpu_freq(f_max)          # 覆盖SAC, 拉满
      record_intervention()        # 写入 buffer 作为负面样本

SAC Trainer (独立进程/线程):
  while True:
    batch = buffer.sample(256)
    update_critic(batch)
    update_actor(batch)
    update_alpha(batch)
    soft_update_target_networks()
    if step % 100 == 0:
      sync_weights_to_actor()      # 热更新
```

### 5.4 TPS 观测窗口

```text
TPS 不要用整个 2s 窗口去平均——那样反应太慢。

实时 TPS = 最近 500ms 内生成了多少 token / 0.5s

具体做法:
  维护一个 token_timestamps deque (记录每个 token 的生成时间)
  
  def get_current_tps():
      now = time.time()
      cutoff = now - 0.5  # 500ms窗口
      recent = [t for t in token_timestamps if t >= cutoff]
      return len(recent) / 0.5

为什么是 500ms:
  - 太短(100ms): 噪声大，几个token的波动就让TPS大幅跳跃
  - 太长(2s): 反应迟钝，频率调整的效果被掩盖在历史数据中
  - 500ms: 适中，既能平滑噪声，又能及时反映频率变化带来的TPS改变

需要注意的是: 对于输出长度短的请求（output=50, 持续约 3s）,
500ms 窗口内可能只生成 3-5 个 token。这种情况下 TPS 读数
的方差很大。SAC 的 Critic 网络中的双 Q 机制会自然地处理
这种高方差——当 TPS 读数不稳定时, Q 值也会不稳定, 从而
降低策略对该特征的信赖。

如果你发现短输出场景下 SAC 训练不稳定, 可以做窗口自适应:
  窗口长度 = max(500ms, 5 × 近期平均 token 延迟)
  但这会增加状态噪声非平稳性, 一般不需要。
```

## 6. SLA Guardian 设计

### 6.1 定位

```
SAC 负责: "在正常情况下找到最优频率"
Guardian 负责: "SAC搞砸了的时候兜底"

类比: 自动驾驶
  - SAC = 自动驾驶策略（可以犯错、可以学习）
  - Guardian = AEB自动紧急制动（不可绕过、纯规则、反应极快）
```

### 6.2 触发条件与行为

```text
触发条件:
  current_tps < 0.85 × tps_slo

行为:
  立即设置 GPU 频率为 f_max（绕过 SAC 输出）
  维持 f_max 直到 TPS 恢复到 SLO 以上 + 额外冷却期 3 秒

为什么是 85%:
  - 100% 太严格: 正常的 TPS 波动（5-10%）是不可避免的
  - 70% 太宽松: 等到 TPS 已经跌了30%才介入，用户体验已受损
  - 85%: 在"正常波动"和"真正违例"之间留出缓冲区

为什么恢复后维持 3 秒冷却期:
  - 防止"拉满→恢复→SAC又降频→又跌破→再拉满"的循环
  - 3秒给系统充足时间稳定，也给Guardian干预后的SAC一个"快照"写入replay buffer
```

### 6.3 防止 Guardian 震荡

**问题**：冷却期结束后 Guardian 交还控制权给 SAC，但如果 SAC 尚未学会"这个状态下不能降频"，它可能立即又输出低频率 → Guardian 再次介入 → 再冷却 → 再交出 → 再介入。形成震荡。

**解决方案**：引入"衰退约束频率上界"（Decaying Frequency Cap）

```text
Guardian 干预后的恢复策略 (不是冷却期后直接交出):

[Guardian干预] → [f_max 维持 3s 冷却期] → [衰退期: 逐步下放频率上限]

冷却期(3s):
  frequency_cap = f_max（完全锁定）

衰退期(15s, 每 2s step 降低一次):
  step 1: frequency_cap = f_max × 0.90
  step 2: frequency_cap = f_max × 0.85
  step 3: frequency_cap = f_max × 0.80
  step 4: frequency_cap = f_max × 0.75
  step 5: frequency_cap = f_max × 0.70
  之后: 取消频率上限, 完全交还给 SAC

最终频率 = min(SAC_target_freq, frequency_cap)  (衰退期内)
```

这样 SAC 从高频状态"被扶着"逐步回到自主决策，不会出现"拉满→缩回去→再拉满"的震荡。同时 SAC 在衰退期内仍可探索，只是被限制了上限。

### 6.4 Guardian 干预的信息利用

```text
Guardian 干预不是浪费——它产生宝贵的训练数据:

干预事件 → 写入 replay buffer 的特殊 entry:
  {
    'state':      干预发生时的状态 (SAC 那个状态下做了错误决策)
    'action':     SAC 的原始动作 (导致问题的动作)
    'actual_freq': f_max (Guardian 实际执行的)
    'reward':     -100 (标记为严重违例)
    'flag':       'GUARDIAN_INTERVENTION'
  }

效果:
  SAC 回放这段经验时会学到:
  "在这个 state 附近不要输出那么低的频率——会被重罚"

衰退期的 step 也写入 buffer (正常 transition, 无 flag):
  这些 transition 中 state 包含了 s₁₄=1 (刚被干预过),
  而 Safe 频率逐渐下降。SAC 从中学到:
  "被 Guardian 干预后, 即使我稍微降频也不会出问题"
  "哪些状态是真的危险, 哪些只是 Guardian 过度保护"
```

同时, 每次 Guardian 干预都会设置 s₁₄=1, 并保持 5 个 step
（10s 内都为 1）。这让 SAC 明确感知"我刚刚被干预过",
做出更保守的决策。

### 6.5 Guardian 的边界

### 6.4 Guardian 的边界

```text
Guardian 不做什么（明确排除）:
  ✗ 不降频 —— 降频是 SAC 的活，Guardian 只负责救火
  ✗ 不学习 —— 纯规则，行为在任何时候都是确定的
  ✗ 不检查温度 —— GPU 有硬件 thermal throttling，软件层面不需要重复
  ✗ 不影响 SAC 训练 —— 只修改实际执行的频率值，不影响 SAC 的参数更新
```

## 7. 负载注入设计

### 7.1 设计目标

负载注入器需要让 SAC 见识足够多样化的场景，以确保学到的策略有泛化能力。具体目标：

1. **覆盖 QPS 从极低到极高的全范围**（让 SAC 知道每个负载级别该用什么频率）
2. **包含突发和震荡模式**（让 SAC 学会预测性提频，而非被动响应）
3. **混合长/短 prompt**（让 SAC 学会区分 Prefill 密集型和 Decode 密集型负载）
4. **每个阶段的 SLO 可以不同**（让 SAC 学会根据 SLO 自适应目标）

### 7.2 请求生成模型

```text
请求生成 = 泊松过程（Poisson Process）

为什么是泊松而非均匀发送:
  - 真实用户请求不是等间隔的——有时密集有时稀疏
  - 泊松的 "无记忆性" 意味着请求之间的间隔不可预测
  - SAC 需要学会处理"前1秒没请求，后1秒突然来10个"的情况

泊松过程的到达间隔:
  inter_arrival_time = -ln(U) / λ
  
  其中 U ~ Uniform(0,1), λ = 目标 QPS
  
  例: λ=4 (QPS), U=0.5
      → inter_arrival = -ln(0.5)/4 = 0.693/4 = 0.173s
```

### 7.3 训练剧本设计

```text
训练剧本 = 多阶段顺序排列，每个阶段有独立的参数

总时长: 约 3.5 小时
阶段间有 30s 过渡（QPS 线性渐变，而非硬切换）

┌──────────┬─────────┬───────────┬──────────┬──────────────┐
│ 阶段      │ 时长    │ QPS       │ Prompt   │ 目标行为     │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 0.基线    │ 1min    │ —         │ —        │ 默认DVFS收据 │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 1.热身    │ 5min    │ 2         │ 中(512)  │ 稳定初始状态 │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 2.低负载  │ 5min    │ 0.5       │ 中(512)  │ SAC学"大胆降频"│
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 3.中负载  │ 10min   │ 4         │ 中(512)  │ SAC学"找甜点" │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 4.高负载  │ 10min   │ 10        │ 中(512)  │ SAC学"不能降" │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 5.突发    │ 10min   │ 1→15→4    │ 中(512)  │ SAC学"预判"   │
│ (3周期)   │         │ (循环)    │          │              │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 6.长prompt│ 5min    │ 2         │ 长(3000) │ 短(50)   │ SAC学"Prefill密集"  │
│ +短输出   │         │           │          │          │ s₀高+s₁低→保高频   │
├──────────┼─────────┼───────────┼──────────┼──────────┼────────────────────┤
│ 7.短prompt│ 5min    │ 3         │ 短(100)  │ 长(1000) │ SAC学"Decode密集"   │
│ +长输出   │         │           │          │          │ s₀低+s₁高→大胆降频  │
├──────────┼─────────┼───────────┼──────────┼──────────┼────────────────────┤
│ 8.高并发+ │ 10min   │ 8         │ 中(512)  │ 中(200)  │ SAC学精细权衡       │
│ 低SLO     │         │           │          │ SLO=30   │ s₁₀(slo达标率)不同  │
├──────────┼─────────┼───────────┼──────────┼──────────────┤
│ 9.正弦震荡│ 10min   │ 1↔8(周期) │ 中(512)  │ SAC学稳定性  │
└──────────┴─────────┴───────────┴──────────┴──────────────┘
```

### 7.4 Prompt 构建方法

```text
必须用 vLLM 实际的 tokenizer 来精确控制 prompt 长度，
因为 tokenizer 的分词方式决定了实际 token 数。

方法:
  1. 准备一段"填充文本"（可以是任何英文/中文段落）
  2. 用 tokenizer(prompt) 计算实际 token 数
  3. 不断拼接填充文本直到 token 数达到目标值
  4. 截断到精确长度

为什么不能靠字符数估算:
  - 英文: ~0.75 token/word (粗略)
  - 中文: ~0.5 token/字符 (粗略)
  - 但不同 tokenizer 差异巨大（Llama vs Qwen vs GPT）
  - 必须用你实际使用的 tokenizer
```

### 7.5 负载注入与 vLLM 的实际连接

```text
负载注入器     HTTP POST /v1/completions      vLLM Server
─────────     ──────────────────────────      ───────────
              {prompt, max_tokens, stream:true}
              
              返回 SSE 流:
              data: {"choices":[{"text":"t1"}]}
              data: {"choices":[{"text":"t2"}]}
              ...每生成一个token返回一行...
              data: [DONE]

在每个 SSE data 行到达时:
  → sac_controller.on_token_generated()
  → 记录时间戳到 token_timestamps deque
```

## 8. 训练方案设计

### 8.1 Replay Buffer 预热（随机策略填充）

这是 off-policy RL 训练的必要前置步骤，没有这一步，SAC 根本无法开始有效学习。

**问题**：SAC 训练的第一步就从 replay buffer 里随机采样一个 batch 来更新 Critic 和 Actor。如果 buffer 是空的（或只有几十条），前几次更新的梯度方向完全随机，甚至可能导致网络参数崩溃。

**解决方案**：在正式训练开始前，先运行一段**"只收集不学习"**的预热阶段。

```text
Phase 0: Buffer 预热（30-60 分钟）
┌────────────────────────────────────────────────────┐
│ 动作策略: 均匀随机策略 a ~ Uniform(-1, 1)           │
│           映射到频率 = 在 [f_min, f_max] 内随机选取 │
│                                                    │
│ SAC 网络: 冻结不更新                                │
│                                                    │
│ 目标: 收集足够多样化的 transition 填满 buffer       │
│       确保 buffer 中覆盖:                           │
│         - 从最低频率到最高频率都有样本               │
│         - 不同负载水平下都有样本                    │
│         - 既有 SLO 满足也有 SLO 违例的样本          │
│                                                    │
│ SLA Guardian: 仍然活跃！                            │
│   Guardian 在预热阶段也正常运行                     │
│   防止随机动作把频率拉太低导致 TPS 崩溃              │
│   Guardian 干预事件也写入 buffer（帮助 SAC 提前      │
│   学习"哪些状态区域是危险的"）                       │
└────────────────────────────────────────────────────┘
```

**为什么是均匀随机而非 SAC 的探索策略**：

| 方式 | 覆盖范围 | 样本多样性 |
|------|---------|-----------|
| **均匀随机** | 全频率空间均匀覆盖 | **最高** — 每个频率被选中的概率相等 |
| SAC + 高 α | 以当前 Actor 的均值为中心 | 中 — 偏离 Actor 太远的区域仍然探索不到 |
| SAC + 低 α | 接近确定性策略 | 低 — 几乎只在当前最好动作附近 |

均匀随机确保 buffer 中**没有盲区**——从 500MHz 到 1980MHz，每个频率区域都有样本。这样 SAC 在开始学习时，不管往哪个方向调整，都有对应的历史经验可以参考。

**需要多少条**：

```text
预热条数目标 = 2250 条 (5 阶段 × 15 min × 30 steps/min)
  
batch_size = 256:
  1 次更新需要 256 条, 2250 条 = 8.8 次更新的独立样本
  MujoCo/Atari 通常需要 10000~50000 frames
  但 GPU 调频的状态空间（15 维）远小于 Atari 画面（84×84×4=28224 维）
  每个 transition 的信息密度远高于像素 → 2250 条可以启动

如果发现 2250 条不够稳:
  方案 A: 延长每个阶段到 30 min → 4500 条, 2.5 小时
  方案 B: 把 step 缩短 500ms (只适用 7B 小模型) → 6750 条, 75 min
  都不需要预热超过 2.5 小时

Replay buffer 可以重复采样, 所以前 2250 条会被反复利用多次。
后续 Phase 1/2/3 会持续填入更多数据, buffer 总量会持续增长。
```

**预热期间的负载注入设计**：

你说得对，预热也需要丰富的样本。如果预热只跑 3 种 QPS、固定 prompt 长度，SAC 训练初期采到的状态空间非常窄——它只见过三种负载水平和单一 prompt 特征，根本不知道"prompt 变长了 GPU 利用率会怎样"、"短 prompt + 长输出时的功耗特征是什么"。

**快速回顾：预热阶段的最终状态空间覆盖目标**

```text
SAC 的 state 是 15 维的, 预热要覆盖的核心差异轴有 3 条:

轴1: QPS/负载水平 (决定 gpu_util, batch_size, queue_depth, power)
     低(1) → 中(5) → 高(10)
     
轴2: LLM推理阶段特征 (决定 gpu_util / mem_util 的比值)
     短prompt+长输出 → 处于 Decode 阶段为主 → gpu_util↓, mem_util↑
     长prompt+短输出 → 处于 Prefill 阶段为主 → gpu_util↑, mem_util↓
     
轴3: 频率范围 (由随机动作自动覆盖)
     500M~1980MHz, 每个频率被选中的概率相等

SAC 需要 buffer 中有各种组合: (低QPS × Prefill × 800MHz), 
(高QPS × Decode × 1500MHz), 等等。

但预热时动作是随机策略——无论处于什么状态, 动作都在全频率
范围内随机。这意味着: 只要状态空间覆盖够广, 每个状态下各种
频率的 transition 会自动填满 buffer。
```

**5阶段预热设计**：

相比之前3阶段(仅QPS分三档)和正式训练的9阶段(精细场景评估)，取中间值：

```text
预热用 5 个阶段, 覆盖 3 条差异轴, 总时长 75 分钟:

┌──────┬──────────┬──────────┬───────────┬─────────────────┬───────┐
│ 阶段  │ 时长     │ QPS      │ prompt    │ 输出长度        │ 负载特征 │
├──────┼──────────┼──────────┼───────────┼─────────────────┼───────┤
│ W1   │ 15min    │ 1 (低)   │ 中(512)   │ 中(200)         │ 低QPS │
│ W2   │ 15min    │ 5 (中)   │ 中(512)   │ 中(200)         │ 中QPS │
│ W3   │ 15min    │ 10(高)   │ 中(512)   │ 中(200)         │ 高QPS │
│ W4   │ 15min    │ 5 (中)   │ 短(100)   │ 长(1500)        │ Decode密集 │
│ W5   │ 15min    │ 5 (中)   │ 长(3000)  │ 短(50)          │ Prefill密集 │
└──────┴──────────┴──────────┴───────────┴─────────────────┴───────┘
```

**输出长度的差异轴**：

W2（output=200）vs W4（output=1500）vs W5（output=50）形成了输出长度的三条对比线：

```text
短输出 (W5, output=50):
  → 请求 3 秒就结束, Decode 窗口短
  → 状态特征: gpu_util 波动大 (频繁进入/退出 Prefill)
  → SAC 在这里学到: "别降太多, 请求马上换一批, 新请求可能不同"

中等输出 (W1/W2/W3, output=200):
  → 请求 12 秒左右, Decode 窗口适中
  → 状态特征: 负载最稳定, 最"正常"的状态
  → SAC 学到: "这是默认基准"

长输出 (W4, output=1500):
  → 请求 90 秒不结束, Decode 窗口极长
  → 状态特征: gpu_util 稳定在低位, mem_util 稳定在高位
  → SAC 学到: 通过 s₁（输出长度）+ s₆（mem_util高）识别出 
    "这是一个长时间稳定的 Decode 状态, 可以大幅降频"
    大幅降频后维持 80+ 秒, 节约大量能量
```

**各阶段的多样性价值**：

```text
W1 (低QPS, 中等prompt):
  ┌────────────────────────────────────────────┐
  │ 状态覆盖: gpu_util=10-25%, batch=1-3,      │
  │          queue≈0, power=低                   │
  │ 价值: "低负载 × 全频率范围"的映射           │
  │       低负载下所有频率都几乎满足SLO          │
  │       SAC学到: 这种情况下降频不吃亏          │
  │ 预计transition: 450条                        │
  └────────────────────────────────────────────┘

W2 (中QPS, 中等prompt):
  ┌────────────────────────────────────────────┐
  │ 状态覆盖: gpu_util=40-65%, batch=4-10,     │
  │          queue=1-5, power=中                 │
  │ 价值: "中等负载 × 全频率范围"的映射        │
  │       这里开始出现SLO分界线:                │
  │       低频(500-800) → TPS可能不够           │
  │       中频(800-1200) → TPS刚好满足          │
  │       高频(1200+) → TPS达标但多余           │
  │ 预计transition: 450条                        │
  └────────────────────────────────────────────┘

W3 (高QPS, 中等prompt):
  ┌────────────────────────────────────────────┐
  │ 状态覆盖: gpu_util=70-90%, batch=15-30+,   │
  │          queue=5-20, power=接近TDP           │
  │ 价值: "高负载 × 全频率范围"的映射          │
  │       ★ 这里产生大量的Guardian干预样本     │
  │       低频(500-800) → TPS暴跌 → Guardian介入 │
  │       中频(800-1200) → 可能违例也可能刚好   │
  │       高频(1200+) → SLO满足但功耗高          │
  │ 预计transition: 450条                        │
  └────────────────────────────────────────────┘

W4 (中QPS, 短prompt+长输出 — Decode密集型 / 长输出关键样本):
  ┌────────────────────────────────────────────┐
  │ 状态覆盖: gpu_util=15-30%, mem_util=80-95%│
  │          (gpu_util ↓ + mem_util ↑ = Decode!)│
  │          batch=4-10, power=中偏低            │
  │ 关键变量: s₁ (output/remaining) = 高值     │
  │          → 状态里明确表示"请求还要待很久"   │
  │ 价值: "长Decode × 全频率范围"的映射        │
  │       ★ output=1500 是输出长度差异轴的极端  │
  │       每个请求 90 秒不结束, 同一频率会运行  │
  │       几十个 step, W4 是这个场景的样本来源   │
  │       SAC需要学到:                          │
  │       1. "当 s₁(输出长度)大 + mem_util高 → │
  │          这是稳定长Decode, 可以大胆降频"    │
  │       2. "降频后 TPS 仅降 5-10%, 功耗降 60%│
  │          × 持续 90 秒 = 巨大能量节省"       │
  │ ★ 这个知识来自W4, W1-W3/W5 都学不到        │
  │ 预计transition: 450条                        │
  └────────────────────────────────────────────┘

W5 (中QPS, 长prompt+短输出 — Prefill密集型 / 短输出对比样本):
  ┌────────────────────────────────────────────┐
  │ 状态覆盖: gpu_util=70-90%, mem_util=50-70%│
  │          (gpu_util ↑ + mem_util ↑ = Prefill!)│
  │          batch=4-10(受batch内长prompt限制)  │
  │          power=高, TTFT敏感                   │
  │ 价值: "Prefill阶段 × 全频率范围"的映射     │
  │       Prefill中, 降频直接增加TTFT           │
  │       SAC需要学到:                          │
  │       "当 gpu_util >> mem_util 且           │
  │        prompt_length 大时, 必须保高频"       │
  │ ★ 这个知识来自W5, W1-W4都学不到             │
  │ 预计transition: 450条                        │
  └────────────────────────────────────────────┘
```

**5阶段 vs 3阶段 vs 9阶段的对比**：

```text
          状态多样性        场景精细度         时长
3阶段:    ⭐⭐ (只覆盖QPS)   ⭐ (粗)          45min
5阶段:    ⭐⭐⭐⭐ (QPS+阶段)  ⭐⭐ (中)        75min  ← 推荐
9阶段:    ⭐⭐⭐⭐⭐ (全覆盖)    ⭐⭐⭐⭐⭐ (精细)  3.5h   ← 正式训练用

3阶段问题: 缺少 W4 和 W5, 预热结束后 buffer 里没有 
           "Decode状态下频率响应"和"Prefill状态下频率响应"的样本
           SAC 开始训练后一遇到 Decode/Prefill 状态就"没见过"
           
5阶段优势: 比3阶段多了两个关键维度(Decode和Prefill的区分),
           但比9阶段少了突发、震荡、高并发低SLO这些只用于
           策略评估而不用于初始状态覆盖的阶段

9阶段的问题: 预热时还不用到9阶段, 因为:
   - 突发/震荡/正弦负载的目的是让 SAC 学会预测性动作
     但预热时动作是随机策略, 它不会"利用超前信息"决定频率
     所以突发阶段的样本对预热没有额外价值——突然高QPS下
     gpu_util=90%的状态和W3的gpu_util=90%状态区别不大
   - 高并发+低SLO: 这修改的是 SLO 值, 不是状态空间
     SLO 可以通过训练时改变配置来覆盖, 不需要预热时覆盖
```

**过渡方式**：

```text
W1 → W2 → W3 → W4 → W5

注意 W3→W4 不只是 QPS 变化, 更是 prompt 长度特征的大幅变化:
  W3: QPS=10, prompt=512 (高负载 + 中等prompt)
  W4: QPS=5,  prompt=100 (中等负载 + 短prompt → Decode密集)

这可能导致 GPU 状态在过渡时剧烈变化。
所以 W3→W4 过渡设为 60s, QPS 从 10 线性降到 5 的同时
prompt 长度从 512 渐变到 100。

过渡期间收集的样本尤其宝贵——它们覆盖了"从高负载中等prompt
向 Decode 密集型过渡"的中间状态, SAC 未来遇到类似场景时
不会觉得"这个状态我没见过"。
```

**预热产出与 buffer 分布检查**：

```text
预热完成(75分钟)后的 buffer:

  总条数: 2250 条 (75min × 30steps/min)
  步长: 2s/step, 所以每阶段约 450条

  按状态空间维度检查分布:

  轴1 (QPS/负载):   低QPS ≈ 20% (W1)
                     中QPS ≈ 60% (W2+W4+W5)
                     高QPS ≈ 20% (W3)
                     → 中负载占比偏高但合理(因为3个阶段在中QPS运行)

  轴2 (阶段特征):    Decode密集 ≈ 20% (W4)
                     Prefill密集 ≈ 20% (W5)
                     混合 ≈ 60% (W1-W3)
                     → 三个类别都有, SAC不会忽视阶段特征

  轴3 (频率范围):    均匀分布 (随机策略保证)
                     [500-800]: ~20%  ← 包含大量Guardian干预
                     [800-1200]:~27%
                     [1200-1500]:~27%
                     [1500-1980]:~27%

  覆盖率检查:
    高QPS+低频 → W3 × 随机到低频 → 大量Guardian干预
    高QPS+高频 → W3 × 随机到高频 → SLO满足但功耗高
    Decode+Pfill → W4/W5 × 全频率 → SAC学会区分阶段的关键数据

  如果检查后发现某个区域稀疏(比如 W4 阶段 GPU 一直空闲,
  batch太小导致样本量不足), 可以单独补跑该负载。
```

**2250条够不够？**

```text
batch_size = 256:

  SAC 采一个 batch = 256 条
  2250/256 ≈ 8.8 次更新的独立样本 (但 replay buffer 可以重复采样)
  buffer 中可以重复采样, 所以 2250 条足够支撑最初的几百次更新

对比常见的 RL 任务:
  Atari 游戏: buffer 预热一般 20000~50000 frames
  MuJoCo 机器人: buffer 预热一般 10000~50000 steps
  但 GPU 调频场景:
    - 状态仅 15 维 (vs Atari 的 28224 维)
    - 奖励反馈每 2s 就能收到 (vs 游戏可能需要数十步)
    - 动作空间仅 1 维连续 (vs 机器人的 6-20 维)
  → 每个 transition 的信息密度高得多 → 2250 条足够启动

如果你觉得 2250 条还不够稳:
  延长每个阶段从 15min 到 30min → 4500 条, 总时长 2.5 小时
  2250 条是"够启动"的下限, 4500 条是"让训练更稳"的推荐
```

**关键**：预热阶段 Guardian 必须开着。均匀随机策略会频繁选中极低频率（如 500MHz），在高负载时这会导致 TPS 暴跌——Guardian 的存在保证"随机乱试"也不会真的搞崩系统。

**预热完成的标志**：
- buffer 中已有 ≥2000 条 transition（75min 预热约 2250 条）
- 且覆盖了至少 3 种不同的负载水平
- 且有 ≥3% 的样本是 Guardian 干预过的（说明探索触碰到了安全边界）

### 8.2 四阶段训练法

完整训练流程包含 Phase 0（Buffer预热）和 Phase 1-3（训练三轮转）：

```text
Phase 0: Replay Buffer 预热（详见 8.1 节）
  ┌──────────────────────────────────────────┐
  │ 目标: 用随机策略填满 buffer，为后续训练   │
  │       提供多样化的初始样本                 │
  │                                          │
  │ 动作: 均匀随机策略 a ~ Uniform(-1, 1)     │
  │ SAC: 冻结不更新                            │
  │ Guardian: 活跃（防止随机动作导致SLO崩溃）  │
  │ 负载: 5 阶段（W1-W5,3QPS+2阶段特征）      │
  │ 时长: 75 分钟 (每个阶段 15min)             │
  │ 产出: ≥2250 条 transition                 │
  └──────────────────────────────────────────┘
         ↓  buffer 填充完成后进入正式训练

Phase 1: Profiling + 离线数据收集（1-2天）
  ┌──────────────────────────────────────────┐
  │ 目标: 获取频率-性能-功耗曲线              │
  │                                          │
  │ 方法:                                     │
  │ 1. 在目标GPU上跑目标模型                  │
  │ 2. 固定频率从500MHz扫到max, 100MHz步进    │
  │ 3. 每个频率跑5分钟, 记录:                │
  │    - 功耗 (W)                             │
  │    - Prefill TTFT (在不同prompt长度下)    │
  │    - Decode TPS (在不同batch size下)      │
  │                                          │
  │ 产出:                                     │
  │ - U型曲线图 (频率 vs 能耗 vs TPS)         │
  │ - Decode 频率无关窗口上界                 │
  │ - Prefill 对不同prompt长度的频率敏感度    │
  │ - 约10K transitions 的初始数据集         │
  └──────────────────────────────────────────┘

Phase 2: 离线预训练（1-2天）
  ┌──────────────────────────────────────────┐
  │ 目标: 让SAC学会不犯致命错误的基本策略     │
  │                                          │
  │ 方法:                                     │
  │ 1. 从Phase 1的profiling数据构建简易模拟器 │
  │ 2. 用CQL (Conservative Q-Learning) 从   │
  │    离线数据中学习，避免外推误差           │
  │ 3. 同时用行为克隆(BC)预训练Actor:        │
  │    L_BC = MSE(SAC_action, expert_action)  │
  │    其中 expert_action 来自一个简单的     │
  │    启发式规则(利用率高→提频，低→降频)     │
  │                                          │
  │ 为什么用CQL而非标准SAC:                   │
  │  离线数据没有探索性动作，标准off-policy   │
  │  RL在离线数据上容易产生外推误差          │
  │  CQL加了一项让Q值在未见过的(state,action) │
  │  上保守估计的正则项                       │
  │                                          │
  │ 产出:                                     │
  │ - 预训练SAC权重 (不会导致严重SLO违例)     │
  └──────────────────────────────────────────┘

Phase 3: 在线微调 + Guardian安全（3-7天）
  ┌──────────────────────────────────────────┐
  │ 目标: 在真实环境中优化到最优               │
  │                                          │
  │ 方法:                                     │
  │ 1. 加载Phase 2的预训练权重                 │
  │ 2. 运行负载注入训练剧本 (3.5小时剧本)       │
  │ 3. Guardian 全程活跃                       │
  │ 4. 初始 f_min_safe = 60% max_freq        │
  │    (限制SAC不能降得太低)                  │
  │ 5. 每当 SLO 达标率 >99% 持续 1 小时，     │
  │    自动下调 f_min_safe 5%                │
  │    (逐步放开安全围栏)                     │
  │ 6. 重复训练剧本2-3轮，确保覆盖不同随机种子 │
  │                                          │
  │ 产出:                                     │
  │ - 生产可用的 SAC 策略                     │
  └──────────────────────────────────────────┘
```

### 8.3 SAC 超参数

| 参数 | 推荐值 | 设计依据 |
|------|--------|---------|
| Actor 学习率 | `1e-4` | 比 Critic 低——策略不能比价值函数变化快，否则训练不稳定 |
| Critic 学习率 | `3e-4` | 需要更快收敛来给 Actor 提供准确的价值估计 |
| α (温度) | 自动调节 | `target_entropy = -1` (单维动作)，α 自动在 [0.01, 1.0] 间调整 |
| γ (折扣) | `0.95` | GPU 调频影响主要在 5-10 步内（10-20秒），不需要更长的远见 |
| τ (Polyak) | `0.005` | 软更新 target 网络，保证训练稳定性 |
| Replay Buffer | `1e6` | 2s/步，一天约 4.3 万条，1M 容量可存约 23 天的数据 |
| Batch Size | `256` | 标准值，平衡梯度稳定性和内存 |
| Actor 网络 | `[128, 128]` | 输入15维→隐藏128→隐藏128→输出2(μ,logσ) |
| Critic 网络 | `[128, 128]` | 输入15+1=16维(state+action)→隐藏128→隐藏128→输出1 |

### 8.4 SAC 网络结构

```text
Actor Network:
  Input(15) → Linear(15,128) → ReLU → Linear(128,128) → ReLU
  → 分叉:
     → Linear(128,1) → μ (mean)
     → Linear(128,1) → clamp → exp → σ (std)
  → Normal(μ, σ).rsample() → tanh → action ∈ [-1,1]

Critic Network (Q1, Q2 结构相同):
  Input(15+1) → Linear(16,128) → ReLU → Linear(128,128) → ReLU
  → Linear(128,1) → Q(s,a) ∈ ℝ

注意:
  - Critic 的输入在第一个 Linear 之前才拼接 action
    (不是在输入层就拼接，而是在第一个隐藏层前)
  - 这样设计让网络先分别学习 state 的特征和 action 的特征
  - log_σ 需要 clamp 到 [-20, 2] 防止数值溢出
```

### 8.5 奖励归一化

```text
SAC 对奖励的绝对值大小敏感。

奖励尺度如果太大（如 ±1000），α 需要调到很大才能匹配
奖励尺度如果太小（如 ±0.01），梯度信号太弱

当前设计的奖励范围 [-100, 10]:
  - 绝对值在两位数量级，对 SAC 友好
  - 不需要额外归一化
  - α 的自动调节能正常工作在这个范围

如果不放心，可以在 SAC 训练循环中加一个 reward scaler:
  reward_scaled = reward / reward_std (用 running estimate of std)
```

## 9. 部署架构

### 9.1 进程拓扑

```text
┌──────────────────────────────────────────────────┐
│                     GPU 节点                      │
│                                                  │
│  ┌─────────────┐   ┌─────────────────────────┐  │
│  │ vLLM Server │   │ SAC Controller (Python)  │  │
│  │ (主进程)    │   │                          │  │
│  │             │   │ ┌──────────────────────┐ │  │
│  │ 推理引擎    │   │ │ 控制线程 (2s心跳)    │ │  │
│  │ /metrics    │◄──│ │ - 状态采集(NVML)    │ │  │
│  │ /v1/comp... │   │ │ - Actor推理(ONNX)   │ │  │
│  └──────┬──────┘   │ │ - GPU调频(NVML)     │ │  │
│         │          │ └──────────────────────┘ │  │
│         │ SSE token│ ┌──────────────────────┐ │  │
│         └──────────┤→│ Guardian线程(100ms)  │ │  │
│                    │ │ - TPS监控            │ │  │
│                    │ │ - 紧急提频           │ │  │
│                    │ └──────────────────────┘ │  │
│                    │ ┌──────────────────────┐ │  │
│                    │ │ 训练线程 (后台)      │ │  │
│                    │ │ - 从ReplayBuffer采样 │ │  │
│                    │ │ - SAC参数更新        │ │  │
│                    │ │ - 定期同步Actor权重  │ │  │
│                    │ └──────────────────────┘ │  │
│                    └─────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

### 9.2 数据流

```text
数据流向:

1. Token 观测流 (每个token触发, ~10-100Hz):
    vLLM SSE → sac.on_token_generated()
   → token_timestamps deque

2. 控制流 (2s一次):
   NVML → GPU状态 → State向量 → SAC Actor(ONNX) → target_freq
   → Guardian检查 → NVML设置频率
   → transition写入Replay Buffer

3. 训练流 (后台持续):
   Replay Buffer → sample(256) → Critic更新 → Actor更新 → α更新
   → 每100步 → 同步Actor权重(ONNX文件)

4. Metrics流 (从vLLM拉取, 2s一次):
   GET /metrics → num_requests_running/waiting
   → state 的 batch_size 和 queue_depth
```

### 9.3 关键技术选择

| 技术点 | 选择 | 理由 |
|--------|------|------|
| Actor 推理格式 | ONNX | <1ms 推理延迟，跨平台，无需PyTorch依赖 |
| 训练框架 | PyTorch | SAC训练需要自动微分，ONNX不适合训练 |
| 权重同步 | 共享文件 + 轮询 | 简单可靠，训练线程写→控制线程读 |
| NVML 绑定 | pynvml | nvidia-ml-py，官方库，所有指标都支持 |
| vLLM 通信 | HTTP/SSE | vLLM 的 OpenAI 兼容接口，无需侵入式修改 |
| Replay Buffer | 内存 + 定期落盘 | 1M 容量约需 200MB 内存，可承受 |

## 10. 实施路线图

### Phase 1: Profiling（第1周）

```text
目标: 获取系统的物理边界

任务:
  □ 在目标 GPU + 模型上完成频率扫频实验
  □ 记录每个频率点的功耗、Prefill TTFT、Decode TPS
  □ 统计不同 prompt 长度下的频率敏感度差异
  □ 产出: U型曲线、Decode频率无关窗口、Prefill最优频率表
  □ 确认: "这个场景的节能上限是多少？"（假设频率随意调）
```

### Phase 2: 离线预训练（第2周）

```text
目标: 产出不犯致命错误的基础策略

任务:
  □ 构建基于 profiling 数据的简易模拟器
  □ 实现启发式专家策略 (利用率→频率的查找表)
  □ 实现 CQL/SAC 离线训练
  □ 用行为克隆预热 Actor
  □ 在模拟器中验证收敛性和策略质量
  □ 产出: 预训练 SAC 权重文件
```

### Phase 3: 在线训练 + Guardian 验证（第3周）

```text
目标: 在真实 vLLM 上完成训练

任务:
  □ 实现负载注入器（训练剧本）
  □ 实现 SAC 控制器（状态采集 + ONNX 推理 + 调频）
  □ 实现 SLA Guardian（100ms 高频监控）
  □ 加载预训练权重，启动在线训练
  □ 运行完整训练剧本 2-3 轮
  □ 监控: 奖励曲线、SLO 违例率、频率分布、功耗曲线
  □ 产出: 训练好的 SAC 策略
```

### Phase 4: A/B 测试（第4周）

```text
目标: 验证 SAC vs 默认 DVFS 的差异

任务:
  □ 用同一组负载 trace 分别跑 SAC 和默认 DVFS
  □ 对比指标:
    - 总能耗 (J)
    - 平均功耗 (W)
    - TPS SLO 达标率
    - P99 TTFT
    - 频率分布图
  □ 显著性检验 (t-test 或 bootstrap)
  □ 产出: A/B 测试报告
```

### 成功标准

| 指标 | 目标 | 当前默认DVFS基准 |
|------|------|-----------------|
| 总能耗降低 | ≥20% | 100% |
| TPS SLO达标率 | ≥99% | 记录基线 |
| P99 TTFT | 不劣于基线+10% | 记录基线 |
| Guardian干预率 | <5% of steps | 记录基线 |
| 频率震荡率 | <20次/分钟 | 记录基线 |

### 失败/转向信号

```text
以下任一出现, 需要重新审视方案:
  □ Phase 2 模拟器中 SAC 策略 SLO 违例率 > 5%
  □ Phase 3 在线训练 2 小时后奖励无上升趋势
  □ Phase 4 Guardian 干预率 > 20% (SAC频繁被覆盖)
  □ Phase 4 能耗降低 < 10% (不够显著)
  □ 频率震荡率 > 60次/分钟 (策略抖动的标志)
```
